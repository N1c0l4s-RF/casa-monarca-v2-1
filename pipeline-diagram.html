🔐 Casa Monarca v2: Pipeline de Autenticación

🔑 FASE 1: Setup Inicial - Generación de Llaves RSA-3072

Cuando un usuario se registra, el sistema genera un par de claves RSA-3072 de forma única y segura.

Generar par RSA-3072

openssl_pkey_new(['private_key_bits' => 3072])

Se generan dos claves complementarias: privada (secreta) y pública (compartible)

Calcular fingerprint

fingerprint = SHA256(clave_publica)

Identificador único de la clave pública para auditoría

Encriptar clave privada

AES-256-CBC + PBKDF2

Salt aleatorio (16 bytes) + IV aleatorio (16 bytes)

Derivación: PBKDF2(usuarioId + salt, 100k iteraciones)

Resultado: encrypted::salt (almacenado en BD)

Almacenar en BD

Campo Valor
clave_privada_encriptada BLOB (AES-256-CBC encriptado)
iv_encriptacion base64(IV)
certificado_publico PEM sin encriptar (es pública)
fingerprint SHA256(clave_publica)
✨ La clave privada NUNCA sale del servidor en forma desencriptada. Solo se usa internamente cuando el usuario emite documentos.
📄 FASE 2A: Crear Documento (Estado: BORRADOR)

POST /api/documentos-create.php

{ "titulo": "Certificado de Participación", "descripcion": "Certificado de participación en el hackathon", "contenido": "Certifico que Juan López participó..." }

Generar folio único

folio = "DOC-" + fecha(Ymd) + "-" + hex_aleatorio(6)

Ejemplo: DOC-20260511-ABC123

Calcular hash SHA-256

hash_sha256 = SHA256(contenido)

Se usa para verificar que el contenido no ha sido alterado

Insertar en BD

Estado: BORRADOR

El documento aún no está firmado

✍️ FASE 3: Emitir Documento (BORRADOR → EMITIDO)

Este es el paso crítico donde se crea la firma digital RSA-3072.

PASO 1: Obtener documento de BD

SELECT folio, contenido, estado FROM documentos WHERE id = ?

PASO 2: Reconstituir mensaje a firmar

⚠️ CRÍTICO: El formato debe ser exactamente igual en emisión y verificación
contenidoFirmar = folio + "|" + contenido Ejemplo: "DOC-20260511-ABC123|Certifico que Juan López participó..."

PASO 3: Desencriptar clave privada

Desencriptación bajo demanda:

Solo cuando el usuario emite (está autenticado)

1. Obtener: encrypted, IV, salt de BD

2. Derivar: clave = PBKDF2(usuarioId + salt, 100k)

3. Desencriptar: AES-256-CBC(encrypted, clave, IV)

4. Resultado: Clave privada PEM en memoria (no persistida)

PASO 4: Firmar con RSA-3072 + SHA-256

openssl_sign( contenidoFirmar, $signature, privateKey, OPENSSL_ALGO_SHA256 ) firma = base64_encode($signature)

Tamaño aproximado: ~512 caracteres en base64

PASO 5: Actualizar documento

UPDATE documentos SET estado = "emitido", firmado_por_usuario_id = usuario_id, fecha_emision = NOW(), firma = base64_encoded_signature WHERE id = ?

PASO 6: Registrar en auditoría

✅ FASE 4: Verificar Documento (PÚBLICO, sin autenticación)

Cualquiera (incluso sin cuenta) puede verificar la validez de un documento usando su folio.

GET /api/consulta_qr.php?token=DOC-20260511-ABC123

Sin autenticación

Obtener documento de BD

SELECT d.*, u.nombre as creador, f.nombre as firmante FROM documentos d JOIN usuarios u ON u.id = d.creado_por LEFT JOIN usuarios f ON f.id = d.firmado_por_usuario_id WHERE d.folio = ?

Validar que esté emitido

✓ Puede verificar

estado = "emitido"

firma ≠ NULL

✗ No puede verificar

estado = "borrador"

estado = "revocado"

Obtener clave pública del firmante

La clave pública NO está encriptada (es pública)

SELECT certificado_publico FROM usuario_claves WHERE usuario_id = ?

Reconstituir mensaje original

⚠️ DEBE ser EXACTAMENTE igual al que se firmó:
contenidoFirmar = folio + "|" + contenido "DOC-20260511-ABC123|Certifico que Juan López participó..."

Verificar firma (RSA-3072 + SHA-256)

openssl_verify( contenidoFirmar, base64_decode(firma_de_bd), publicKey, OPENSSL_ALGO_SHA256 ) Resultado: 1 = Firma válida ✓ 0 = Firma inválida ✗ -1 = Error en verificación

Devolver respuesta

{ "documento": { "folio": "DOC-20260511-ABC123", "titulo": "Certificado", "estado": "emitido", "creador": "Juan López", "firmante": "Juan López", "fecha_emision": "2026-05-11 10:32:00" }, "verificacion": { "firma_valida": true, "mensaje": "Firma digital válida ✓" } }
🌐 Este endpoint es público y puede ser llamado desde un navegador, una app móvil, o incluso un código QR. No requiere credenciales.

📊 Flujo Completo Resumido
El flujo completo de un documento:

1️⃣ SETUP: Usuario se registra → RSA-3072 generado y encriptado en BD

2️⃣ CREAR: Documento en estado BORRADOR, sin firma

3️⃣ EMITIR: Clave privada desencriptada → Firma RSA-3072+SHA256 → Guardada en BD

4️⃣ VERIFICAR: Público, sin auth → Obtiene folio → Obtiene clave pública → Verifica firma

5️⃣ REVOCAR: Marca como revocado → Se mantiene para auditoría → La verificación pública lo muestra

Puntos Críticos:

  • ✅ Formato de firma: folio + "|" + contenido debe ser idéntico en emisión y verificación
  • ✅ Clave privada: Solo en memoria durante emisión, nunca en tránsito desencriptada
  • ✅ Verificación pública: No requiere autenticación, pero verifica criptográficamente
  • ✅ Auditoría: Cada acción registrada (creación, emisión, revocación, verificación)
  • ✅ Revocación: No elimina el registro, lo marca para trazabilidad
